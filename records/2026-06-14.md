# 2026-06-14 Docker 化部署记录

## 背景

项目已经从内存版 `MessageStore` 切换到 Redis 持久化版，服务端运行依赖 Redis、Muduo、Protobuf、hiredis 和 C++ 运行环境。

本机开发环境已经安装过 Muduo，所以宿主机可以直接通过 CMake 编译；但是 Docker build 是隔离环境，默认看不到宿主机的 `/usr/local/lib/libmuduo*.a` 和 `/usr/local/include/muduo`，因此 Dockerfile 必须在镜像内部重新准备 Muduo。

## Docker 实现

当前采用两个镜像、两个容器的部署方式：

```text
server 镜像
  reliable-message-delivery:local
  包含 /app/bin/server、/app/bin/client、Muduo、Protobuf、hiredis 等运行依赖

redis 镜像
  docker.m.daocloud.io/library/redis:7-alpine
  提供 Redis 服务和 redis-cli
```

`docker-compose.yml` 不会把 server 和 Redis 打成一个最终镜像，而是负责编排两个容器一起运行：

```text
server 容器  --Docker Compose 内部网络-->  redis 容器
```

server 通过环境变量连接 Redis 容器：

```text
RMD_REDIS_HOST=redis
RMD_REDIS_PORT=6379
```

这里的 `redis` 是 Compose 服务名，会被 Docker 内部 DNS 解析到 Redis 容器 IP，不是宿主机的 `127.0.0.1`。

## Dockerfile 构建逻辑

当前 Dockerfile 使用多阶段构建：

```text
build 阶段
  1. 使用 DaoCloud Debian 基础镜像
  2. 将 apt 源切换到 USTC Debian 镜像源
  3. 安装 build-essential、cmake、git、boost、protobuf、hiredis 等基础依赖
  4. 从 https://github.com/chenshuo/muduo.git 拉取 Muduo 源码
  5. 执行 INSTALL_DIR=/usr/local ./build.sh install 编译并安装 Muduo
  6. 编译 ReliableMessageDelivery 的 server/client

runtime 阶段
  1. 使用干净的 Debian runtime 镜像
  2. 安装运行时依赖
  3. 从 build 阶段复制 /usr/local 和 /src/bin/server、/src/bin/client
  4. 默认启动 /app/bin/server
```

最初尝试在 Dockerfile 中使用：

```bash
apt-get install libmuduo-dev
```

但 Debian bookworm 默认仓库没有 `libmuduo-dev`，构建失败：

```text
E: Unable to locate package libmuduo-dev
```

因此改为在 Dockerfile 内部源码编译 Muduo。

## 当前错误逻辑：登录触发大量 Pending 补发导致客户端超时

Redis 中发现 `uid=1` 有大量 Pending 消息：

```redis
ZCARD rmd:pending:1
(integer) 371988
```

压测客户端模型中 receiver 固定使用：

```text
uid=1
```

客户端启动流程是：

```text
1. receiver 登录 uid=1
2. sender 登录 uid=1000、uid=1001...
3. sender 开始发送 CHAT_REQ
```

服务端登录处理逻辑当前是同步执行：

```text
handleLogin()
  -> bindUser()
  -> send LOGIN_RESP
  -> deliverPendingMessages(uid)
```

`deliverPendingMessages(uid)` 当前会读取该用户所有 Pending 消息并同步补发：

```text
getPendingMessages(uid)
for each pending record:
  tryDeliverMessage(record)
```

问题在于：登录、聊天、ACK、连接关闭事件都由同一个 `worker_` 顺序消费。虽然服务端还有一个 `delivery_worker_`，但它只负责扫描 Delivered 超时消息并重投，不负责处理登录队列。

因此当 receiver `uid=1` 登录后，`worker_` 会进入大量历史 Pending 补发流程。在这个流程返回之前，后续 sender 的 `LOGIN_REQ` 虽然已经被网络线程收到并入队，但不会被业务 worker 及时处理。

日志表现为：

```text
收到 uid=1000 LOGIN_REQ
客户端等待 LOGIN_RESP 超时并断开
几十秒后服务端才打印 User online: uid=1000
```

客户端侧表现为：

```text
failed to start sender uid=1000
```

这不是 sender 登录包没有到达服务端，而是到达后排在业务队列里，等待前面的 receiver Pending 补发完成。

## 为什么看起来“一条也发不出去”

有两个关键原因：

```text
1. getPendingMessages(uid) 可能先从 Redis 一次性读取几十万条 Pending 记录，读取阶段就可能很久。
2. 即使开始发送 CHAT_PUSH，接收方 ACK 也要回到同一个 worker_ 队列处理；worker_ 正在补发循环中，ACK 无法及时被消费。
```

客户端必须先收到 `LOGIN_RESP`，才认为登录成功并继续后续流程。sender 登录响应被延迟超过客户端 10 秒等待时间后，sender 会失败退出。receiver 后续也可能被 stop，导致大量 Pending 没有真正进入 Acked。

## 当前结论

当前问题不是 Docker 问题，也不是 Redis 连接问题，而是服务端业务模型中的同步全量补发问题：

```text
用户登录
  -> 同步全量补发 Pending
  -> 阻塞唯一业务 worker
  -> 后续登录/聊天/ACK 延迟
  -> 客户端登录超时
  -> Pending 继续积压
```

## 后续修复方向

`deliverPendingMessages()` 不应该在登录处理函数里同步全量补发。后续应改为：

```text
1. 登录只负责绑定用户并尽快返回 LOGIN_RESP。
2. Pending 补发改为异步任务。
3. 每次只补发固定数量，例如 100 或 1000 条。
4. 补发批次之间让出 worker，使 LOGIN、CHAT、ACK 可以继续被处理。
5. 对 Pending 补发增加限流、分页和可观测指标。
```

更合理的目标形态：

```text
handleLogin()
  -> bindUser()
  -> send LOGIN_RESP
  -> enqueuePendingDelivery(uid)

pending delivery worker
  -> 每轮读取一小批 Pending
  -> tryDeliverMessage()
  -> 等待 ACK 或下一轮继续补发
```

这样可以避免单个用户的大量历史消息拖垮整个业务 worker。
